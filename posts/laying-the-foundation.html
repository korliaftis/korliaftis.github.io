Laying the Foundation

January 19, 2025

Since I started working with computers I always loved the idea of a homelab, a safe space to mess with stuff, break stuff and sometimes learn a few things along the way.

Back when I was trying to land my first tech job my homelab was a bunch of noisy Cisco switches and routers. Along the way I discovered GNS3 and eventually those physical switches and routers were virtualized on my beloved Sony Vaio VGN-N31Z with it’s Core 2 Duo, 2GB of RAM and 120GB of spinning rust.

A few years later when I started working with Windows my homelab transitioned into a Windows Server 2012 R2 VM running on VirtualBox. It didn’t take long to realize that if I wanted to run more than just a couple of VMs at the same time I would need some serious hardware. So I ended up breaking the bank on a desktop running the free version of VMware ESXi packed with an Intel i7, 64GB of RAM and a bunch of SSDs.

Fast forward again and with everything moving to the public cloud my homelab shifted to an AWS account. What was once a capital expense with finite resources became an operational expense with “limitless” capabilities. However, this newfound flexibility quickly turned into a source of stress thanks to its ever-increasing fees and charges.

These days my homelab is a “vintage” Dell Latitude with an Intel i3, 16GB of RAM and a 1TB SSD running Debian 12.

Foundation

This guide begins after you have selected the hardware for your homelab, downloaded a Debian 12 ISO, created a bootable USB and completed a minimal installation with only the “standard system utilities” and “SSH server” selected. By now you should know your machine’s IP address on the network and have successfully logged in via SSH.

Let’s get started..

  • switch to the root user..

    su -

  • set the hostname..

    HOSTNAME="server"

hostnamectl set-hostname ${HOSTNAME}

  • set the timezone..

    TIMEZONE_REGION="Europe"
TIMEZONE_CITY="Athens"

timedatectl set-timezone ${TIMEZONE_REGION}/${TIMEZONE_CITY}

  • disable the cdrom source (if we used the DVD/USB Debian image)..

    sed -i '/^#/! {/cdrom/ s/^/# /}' /etc/apt/sources.list

  • set the default locale..

    apt install -y locales-all
echo 'LC_ALL="en_US.UTF-8"' >> /etc/default/locale

  • install packages..

    apt update && apt upgrade -y
apt install -y bash-completion ca-certificates apt-transport-https vim git curl jq gnupg unzip htop tree tmux nmap tcpdump ethtool strace

  • configure sudo..

    USERNAME="user"

apt install -y sudo
usermod -a -G sudo ${USERNAME}

  • enable sudo without password..

    USERNAME="user"

echo "${USERNAME} ALL=(ALL) NOPASSWD:ALL" | EDITOR='tee -a' visudo
visudo --check

  • disable the laptop lid function (if we use a laptop)..

    echo 'HandleLidSwitch=ignore' >> /etc/systemd/logind.conf

  • suppress the output of kernel logs on the console..

    echo 'kernel.printk = 1 4 1 7' >> /etc/sysctl.conf
sysctl -p

  • increase the /proc/sys/fs/inotify/max_user_instances value..

    echo 'fs.inotify.max_user_instances = 16384' >> /etc/sysctl.conf
sysctl -p

  • change the way the bash prompt looks..

    USERNAME="user"

echo 'export PS1="\[\e[37m\][\[\e[93m\]\w\[\e[37m\]] \[\e[0m\]"' >> /home/${USERNAME}/.bashrc
echo 'export PS1="\[\e[37m\][\[\e[91m\]\w\[\e[37m\]] \[\e[0m\]"' >> /root/.bashrc

  • create a few files and directories to keep our stuff organized..

    USERNAME="user"

su - ${USERNAME} -c "touch /home/${USERNAME}/.bash_aliases"
su - ${USERNAME} -c "mkdir /home/${USERNAME}/.project"
su - ${USERNAME} -c "mkdir /home/${USERNAME}/.ssl"
su - ${USERNAME} -c "mkdir /home/${USERNAME}/.scripts"
echo 'export PATH="$PATH:/home/user/.scripts"' >> /home/${USERNAME}/.bashrc

  • create an ssh key for our user account..

    USERNAME="user"

su - ${USERNAME} -c "ssh-keygen -t rsa -b 4096"
su - ${USERNAME} -c "touch /home/${USERNAME}/.ssh/config"
chmod 600 /home/${USERNAME}/.ssh/config

  • update the password of our user account..

    USERNAME="user"

passwd ${USERNAME}

  • update the password of the root account..

    passwd root

Building Blocks

Now that we’ve established the foundation, we can begin adding our building blocks.

Since I work as a Site Reliability Engineer the tools I rely on include the following..

  • docker

    USERNAME="user"

curl -L https://download.docker.com/linux/debian/gpg | gpg --dearmor > /etc/apt/keyrings/docker.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/debian $(. /etc/os-release && echo "$VERSION_CODENAME") stable" > /etc/apt/sources.list.d/docker.list
apt update && apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
usermod -aG docker ${USERNAME}
systemctl daemon-reload
systemctl enable docker.service
systemctl start docker.service

    By default Docker binds published ports to all interfaces (0.0.0.0) making containers accessible from anywhere. I prefer the opposite, containers should be private unless explicitly exposed.

    systemctl stop docker.service
cat << 'EOF' > /etc/docker/daemon.json
{
  "ip": "127.0.0.1",
  "metrics-addr": "0.0.0.0:9323"
}
EOF
systemctl start docker.service

  • kubectl

    curl -Lo /tmp/kubectl "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
install -o root -g root -m 0755 /tmp/kubectl /usr/local/bin/kubectl
kubectl completion bash > /etc/bash_completion.d/kubectl

  • k9s

    TEMP_DIR="$(mktemp -d)"
SYSTEM_ARCH="$(dpkg --print-architecture)"
K9S_VERSION="$(curl -s https://api.github.com/repos/derailed/k9s/releases/latest | jq -r .tag_name)"

curl -sLo ${TEMP_DIR}/k9s_Linux_${SYSTEM_ARCH}.tar.gz https://github.com/derailed/k9s/releases/download/${K9S_VERSION}/k9s_Linux_${SYSTEM_ARCH}.tar.gz
tar -xzf ${TEMP_DIR}/k9s_Linux_${SYSTEM_ARCH}.tar.gz --directory ${TEMP_DIR}
cp ${TEMP_DIR}/k9s /usr/local/bin/k9s
k9s completion bash > /etc/bash_completion.d/k9s

  • minikube

    curl -Lo /tmp/minikube_latest_amd64.deb https://storage.googleapis.com/minikube/releases/latest/minikube_latest_amd64.deb
dpkg -i /tmp/minikube_latest_amd64.deb
minikube completion bash > /etc/bash_completion.d/minikube

    When Minikube starts it checks for a Docker network with the same name and creates it automatically if it doesn’t exist. By creating this network manually we can define a custom IP range making the node IPs predictable.

    docker network create "minikube" --driver="bridge" --subnet="10.20.30.0/24" --gateway="10.20.30.1"

  • haproxy

    A simple reverse proxy to easily access HTTP workloads running on Minikube. It listens on ports 80 and 443 of the system and forwards connections to the Minikube network.

    apt install -y haproxy
systemctl daemon-reload
systemctl enable haproxy.service
systemctl stop haproxy.service
cp /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg.bak
cat << 'EOF' > /etc/haproxy/haproxy.cfg
global
  log stdout format raw local0 info

defaults
  log global
  timeout client 30000
  timeout connect 5000
  timeout server 30000

frontend stats
  mode http
  bind *:8404
  stats enable
  stats uri /
  stats refresh 30s
  no log

frontend metrics
  mode http
  bind *:8405
  http-request use-service prometheus-exporter
  no log

frontend haproxy-http
  mode tcp
  bind *:80
  default_backend minikube-http
  option tcplog

frontend haproxy-https
  mode tcp
  bind *:443
  default_backend minikube-https
  option tcplog

backend minikube-http
  mode tcp
  server minikube 10.20.30.2:31080 check
  server minikube-m02 10.20.30.3:31080 check
  server minikube-m03 10.20.30.4:31080 check
  server minikube-m04 10.20.30.5:31080 check
  server minikube-m05 10.20.30.6:31080 check

backend minikube-https
  mode tcp
  server minikube 10.20.30.2:31443 check
  server minikube-m02 10.20.30.3:31443 check
  server minikube-m03 10.20.30.4:31443 check
  server minikube-m04 10.20.30.5:31443 check
  server minikube-m05 10.20.30.6:31443 check
EOF
haproxy -c -f /etc/haproxy/haproxy.cfg
systemctl start haproxy.service

  • helm

    curl -L https://baltocdn.com/helm/signing.asc | gpg --dearmor > /etc/apt/keyrings/helm.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/helm.gpg] https://baltocdn.com/helm/stable/debian/ all main" > /etc/apt/sources.list.d/helm.list
apt update && apt install -y helm
helm completion bash > /etc/bash_completion.d/helm

  • terraform

    USERNAME="user"

curl -L https://apt.releases.hashicorp.com/gpg | gpg --dearmor > /etc/apt/keyrings/hashicorp.gpg
echo "deb [signed-by=/etc/apt/keyrings/hashicorp.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" > /etc/apt/sources.list.d/hashicorp.list
apt update && apt install -y terraform
echo 'complete -C /usr/bin/terraform terraform' >> /home/${USERNAME}/.bashrc

Ongoing Maintenance

Setting things up once is fine but without updates we are stuck with a “snapshot” of the system from the day it was installed.

Running apt update && apt upgrade -y regularly will keep both the system and most of the tools up to date.

To update kubectl, k9s and minikube we can simply run their installation commands again.

Tactical Enhancements

This part of the guide covers things that are nice to have but not strictly necessary.

A nice upgrade for our homelab is registering a domain, it doesn’t matter if it’s a .com or a .xyz. With a domain we can set up DNS records that work consistently across devices, like A records (domain.xyz and *.domain.xyz) pointing to our homelab’s internal IP. We can also use Cloudflare’s free tier to manage DNS and take advantage of it’s API and with an ACME client issue wildcard TLS certificates for our domain.

I use acme.sh for this. By providing my Cloudflare Zone ID, Account ID and API Token along with the domain name a valid certificate is generated and saved to my ~/.ssl directory.

DOMAIN_NAME=""
CF_ZONE_ID=""
CF_ACCOUNT_ID=""
CF_TOKEN=""

USERNAME="user"
TEMP_DIR="$(mktemp -d)"

docker run \
  --rm \
  --interactive \
  --tty \
  --env CF_Zone_ID="${CF_ZONE_ID}" \
  --env CF_Account_ID="${CF_ACCOUNT_ID}" \
  --env CF_Token="${CF_TOKEN}" \
  --volume ${TEMP_DIR}/ssl:/acme.sh \
  neilpang/acme.sh:latest \
  acme.sh \
  --dns dns_cf \
  --domain ${DOMAIN_NAME} \
  --domain *.${DOMAIN_NAME} \
  --issue \
  --keylength 4096 \
  --server letsencrypt

cp -r ${TEMP_DIR}/ssl/${DOMAIN_NAME}/* /home/${USERNAME}/.ssl
chown -R ${USERNAME}:${USERNAME} /home/${USERNAME}/.ssl/*
rm -rf ${TEMP_DIR}
history -c && history -w

Another useful upgrade for our homelab is setting up remote access. We could rent a VPS and set up WireGuard tunnels between our devices but this adds latency. A better option is using something like Nebula where the VPS can act as a broker and not as a relay. However, the simplest choice is to use a managed services like Tailscale or ZeroTier which will handle everything for us.

The last upgrade to consider is configuring wake-on-lan for our homelab to power it on remotely. Most systems support it, we just need to enable it in the BIOS. Pair wake-on-lan with an always-on Raspberry Pi on the same network and a remote access solution like the ones mentioned above and we will be able to connect to our homelab from anywhere, anytime.

Closing Thoughts

Allright, we saw a bunch of things and touched many topics but one thing we did not cover is security.

So is it secure? Well, we prioritized convenience over security in a few areas but at the end of the day this is a homelab not a production system, it will never store sensitive data and if someone breaks into my house to steal my stuff my trusty old Dell Latitude will be the least of my concerns.

At the end of the day this is all about experimentation and learning so take what works for you and make it your own.
